# ACTV Validation Data  
### Cleaning Pipeline and Interactive Prototype

This repository provides a **reproducible Python workflow** for cleaning ACTV ticket
validation data and an **interactive Dash prototype** for exploring the processed dataset.

The project is designed to ensure **transparency, reproducibility, and reusability**
of the data preparation steps, and to support exploratory **spatiotemporal analyses**
of ticket validation patterns.



## Repository overview

The repository is organised into two main components:

1. **Data cleaning pipeline**  
   A modular Python pipeline that transforms raw ACTV ticket validation data into a
   **spatially consistent, de-duplicated, and semantically categorised dataset**.

2. **Interactive prototype**  
   A Dash web application for exploring the processed data by date range, ticket type,
   stop location, and time of day.

The prototype operates exclusively on the output generated by the cleaning pipeline.


## Data cleaning pipeline

The data cleaning pipeline performs the following steps:

### 1. Stop harmonisation
- Water stops are preserved as provided in the raw data.
- Land stops are mapped to a predefined set of **land key areas**.
- Each validation record is enriched with:
  - `stop_name`: name of the transport stop;
  - `latitude`: latitude coordinate of the stop;
  - `longitude`: longitude coordinate of the stop.

This step ensures spatial consistency across heterogeneous stop definitions.


### 2. Temporal de-duplication
- Repeated validations associated with the same `(serial, stop)` within a fixed time
  window are removed.
- Default window: **5 minutes**.
- Only the first validation within each window is retained.

This step mitigates biases introduced by repeated validations at gates or onboard
systems.


### 3. Ticket categorisation
- A semantic `ticket_code` is assigned using the `title_description` field.
- Matching is case-insensitive and based on exact lists with light text normalisation.
- Tickets are grouped into:
  - short-term and multi-day tickets;
  - monthly and yearly subscriptions;
  - specific user categories (e.g. students, workers, retirees, residents).


### 4. Filtering of unclassified records
- Records without a valid `ticket_code` are removed.
- Summary statistics are produced, including:
  - number and percentage of removed records;
  - affected ticket descriptions;
  - occurrence counts.




## Interactive prototype

The `prototype/` folder contains a Dash web application that allows interactive exploration of the processed dataset.

The prototype supports:
- selection of arbitrary date ranges (with missing dates automatically disabled);
- filtering by ticket type;
- spatial visualisation of validation counts by stop;
- temporal aggregation of validations by hourly time slots.

The prototype is intended for **exploratory analysis and visual inspection** and does not perform any additional data cleaning.

## Code execution
### 1. Setup
First clone the repository and install the requirements. 
```text
# clone
git clone https://github.com/giuRov/mobilityVenice.git
cd mobilityVenice

# install dependencies
pip install -r requirements.txt
```

### 2. Data placement
This repository does not version raw/processed data. Download the datasets from Zenodo and place them as follows:
```text
mobilityVenice/
└─ data/
   └─ raw/
      ├─ <your_raw_files_here>
      └─ ...
```
See `data/README.md` for the expected filenames and formats.

### 3. Run the cleaning pipeline
Example run via the helper script (for the file `winter.csv`):
```text
python scripts/run_pipeline.py \
  --raw-dir data/raw \
  --out-dir data/processed \
  --validations winter.csv \
  --dedup-minutes 5 \ 
  --output-prefix outputFile 
```
The pipeline performs temporal de-duplication (5-minute window by default), aggregation of land transport stops, ticket categorisation, and filtering of unclassified records.
The de-duplication window can be adjusted via `--dedup-minutes`.  
The name of the processed output can be customised using `--output-prefix`; if not specified, the output filename is automatically derived from the input file name by appending the `Processed` suffix (e.g. `winter.csv` becomes `winterProcessed.csv`). 


### 4. Run the Dash prototype
The prototype reads only the processed output generated by the pipeline. From the repository root, run:
```text
python -m prototype.app \
  --data data/processed/winterProcessed.csv
```
If your stop files are located elsewhere, you can provide them manually:
```text
python -m prototype.app \
  --data data/processed/winterProcessed.csv \
  --stops-water data/raw/stopsWater.csv \
  --stops-land-mapped data/raw/stopsLandMapped.csv \
  --land-key-areas data/raw/landKeyAreas.csv
```

Then open the local URL shown in the terminal (typically `http://127.0.0.1:8050/`).
