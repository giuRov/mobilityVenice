# Smart Card Data in Venice 
### Cleaning Pipeline and Interactive Prototype

This repository provides a **reproducible Python workflow** for processing ACTV ticket
validation data and an **interactive Dash prototype** for exploring the processed dataset.

The project is designed to ensure **transparency, reproducibility, and reusability**
of the data preparation steps, and to support exploratory **spatiotemporal analyses**
of ticket validation patterns.



## Repository overview

The repository is organised into two main components:

1. **Processing pipeline**  
   A modular Python pipeline that transforms raw ACTV ticket validation data to obtain
   cleaned and enriched ticket validation data suitable for analysing mobility patterns
   in the historic city of Venice, with particular emphasis on the waterborne transport
   network.

3. **Interactive prototype**  
   A Dash web application for exploring the processed data by date range, ticket type,
   user category. 

The prototype operates exclusively on the output generated by the processing pipeline.


## Data processing pipeline

The data processing pipeline performs the following steps:

### 1. Data Cleaning
- Repeated validations associated with the same `(serial, stop_id)` within a fixed time
  window are removed.
- Default window: **5 minutes**.
- Only the first validation within each window is retained.

This step mitigates biases introduced by repeated validations at gates or onboard
systems.

### 2. Spatial Aggregation of Land Transport Stops
- Water stops are preserved as provided in the raw data.
- Land stops are mapped to a predefined set of **land key areas**.


### 3. Ticket-Based User Categorisation
- Aggregation of multiple ticket types to define semantically homogeneous ticket classes (`ticket_class`) based on the `title_description` field.
- The obtained dictionary (`ticket_class`, `title_description`) is used to define the relevant user categories. The matching is case-insensitive. 
- Ticket classes with their identifiers are: 
  - 75-minute ticket (75);
  - one-day ticket (D-1);
  - two-day ticket (D-2);
  - three-day ticket (D-3);
  - seven-day ticket (D-7);
  - student monthly ticket (M-STUD);
  - student yearly ticket (Y-STUD);
  - retiree ticket (RET);
  - monthly ticket (M-RES);
  - yearly ticket (Y-RES).

- User categories are:
  - Occasional travellers;
  - Toursits;
  - Students;
  - Retirees;
  - Residents.


### 4. Ticket Filtering
- Records not belonging to a valid `ticket_class` are removed.


The data processing pipeline generates descriptive statistics on the processed data, including:
  - number and percentage of removed records;
  - removed titles and number of removed validations for each title.




## Interactive prototype

The `prototype/` folder contains a Dash web application that allows interactive exploration of the processed dataset.

The prototype supports:
- selection of arbitrary date ranges (with missing dates automatically disabled);
- filtering by ticket type;
- filtering by user category;
- spatial visualisation of validation counts per stop;
- temporal aggregation of validations by hour. 

The prototype is intended for **exploratory analysis and visual inspection** and does not perform any additional data cleaning.

## Code execution
### 1. Setup
First clone the repository and install the requirements. 
```text
# clone
git clone https://github.com/giuRov/mobilityVenice.git
cd mobilityVenice

# install dependencies
pip install -r requirements.txt
```

### 2. Data placement
This repository does not version raw/processed data. Download the datasets from Zenodo and place them as follows:
```text
mobilityVenice/
└─ data/
   └─ raw/
      ├─ <your_raw_files_here>
      └─ ...
```
See `data/README.md` for the expected filenames and formats.

### 3. Run the cleaning pipeline
Example run via the helper script (for the file `winter_raw.csv`):
```text
python scripts/run_pipeline.py \
  --raw-dir data/raw \
  --out-dir data/processed \
  --validations winter_raw.csv \
  --dedup-minutes 5 \ 
  --output-prefix outputFile 
```
The pipeline performs data cleaning (5-minute window by default), spatial aggregation of land transport stops, ticket categorisation, and filtering of unclassified records.
The de-duplication window can be adjusted via `--dedup-minutes`.  
The name of the processed output can be customised using `--output-prefix`; if not specified, the output filename is automatically derived from the input file name by removing the `raw` suffix (e.g. `winter_raw.csv` becomes `winter.csv`). 


### 4. Run the Dash prototype
The prototype reads only the processed output generated by the pipeline. From the repository root, run:
```text
python -m prototype.app \
  --data data/processed/winter.csv
```
If your stop files are located elsewhere, you can provide them manually:
```text
python -m prototype.app \
  --data data/processed/winter.csv \
  --stops-water data/raw/stopsWater.csv \
  --stops-land-mapped data/raw/stopsLandMapped.csv \
  --land-key-areas data/raw/landKeyAreas.csv
```

Then open the local URL shown in the terminal (typically `http://127.0.0.1:8050/`).
